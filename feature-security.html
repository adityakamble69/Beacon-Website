// FEATURE — SECURITY

Your code stays yours,
at every layer.

Encrypted at rest and in transit, scoped by access control, and never used to train models — the same repo you'd hand to a new hire under NDA, not a third party.

What's actually enforced

Security claims on a landing page are easy to make and hard to verify, so Beacon's SOC 2 report and a signed DPA are available on request before you connect a single repo — not just after a sales call.

  • Source code and docs are encrypted at rest with per-tenant keys, and in transit via TLS 1.3
  • Access is scoped per repo and per seat, with SSO and audit logs on the Enterprise tier
  • Indexed content is never used to train or fine-tune any model, Beacon's or anyone else's

Why teams care

Security review is usually the step that kills a dev-tool trial, not the product itself. Making the posture legible up front — what's encrypted, who can see what, where the data does and doesn't go — turns that review into a checklist instead of a blocker.

Where it fits

Regulated industries Fintech and healthtech teams get a SOC 2 report and DPA before repo access is even granted.
Enterprise procurement SSO and per-repo access scoping mean security review doesn't stall the rollout.
Open-source-adjacent teams Private forks and internal-only docs stay off any training set, by policy and by architecture.
